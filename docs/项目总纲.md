# MiniOS 总规划：基于 myCPU RISC-V 模拟器的最小操作系统演进路线

## 一、项目定位

MiniOS 是建立在 myCPU RISC-V 模拟器之上的最小操作系统实验项目。

本项目不是一开始就追求完整的现代操作系统，而是希望通过一个可控、可解释、可测试的最小系统，把计算机组成原理、操作系统、RISC-V 特权架构和系统软件开发串联起来。

当前 myCPU 项目已经具备 RISC-V 指令模拟器的核心骨架，包括 CPU、DRAM、Bus、CSR、Exception 以及 UART、CLINT、PLIC 等模块。MiniOS 的目标是在此基础上，逐步让模拟器从“能执行指令”演进到“能承载一个最小内核运行”。

因此，本项目的长期方向是：

> 从 RISC-V 指令集模拟器出发，逐步构建一个能够启动、输出、处理中断、管理内存、执行简单任务切换和系统调用的 MiniOS。

---

## 二、为什么要做 OS 部分

CPU 模拟器解决的是“指令如何被执行”的问题，而 OS 部分解决的是“一个程序如何在机器之上组织运行”的问题。

如果只有 CPU 模拟器，项目主要停留在：

- 取指
- 译码
- 执行
- 访存
- 异常处理
- CSR 读写
- 外设寄存器模拟

这些内容已经能够支撑组成原理和 RISC-V 指令集学习。

但如果希望进一步理解操作系统，就必须继续回答：

- 内核代码如何被加载？
- CPU 复位后如何进入内核入口？
- 内核如何设置栈？
- 内核如何使用串口输出？
- 异常和中断如何进入 trap handler？
- 定时器中断如何触发调度？
- 系统调用如何从用户态进入内核态？
- 多个任务如何共享 CPU？
- 虚拟内存和页表如何改变地址访问？

MiniOS 的意义就在于，把这些问题从课本概念变成可以运行、可以调试、可以测试的系统。

## 三、总体建设原则

### 1. 先裸机，后操作系统

不要一开始就追求进程、文件系统、虚拟内存和复杂调度。

第一阶段目标只是让一段裸机程序能够在模拟器上运行，并通过 UART 输出字符。

### 2. 先 M 模式，后 S 模式

RISC-V 特权架构中，M 模式是最高特权级。项目初期可以先让内核运行在 M 模式，降低复杂度。

等 M 模式下的启动、异常、中断、UART 输出稳定后，再考虑引入 S 模式，把 MiniOS 逐步做得更接近真实操作系统。

### 3. 先物理内存，后虚拟内存

当前模拟器已经有 DRAM 模型。MiniOS 初期直接使用物理地址即可。

页表、SATP、地址翻译、TLB、页异常等内容放到后期，不要提前引入。

### 4. 先单任务，后多任务

先让一个 kernel_main 正常运行，再做简单任务结构。

不要在内核还不能稳定启动时就写调度器。

### 5. 先轮询，后中断

UART 输出、定时器读取等可以先用轮询方式实现。

等 Bus、CLINT、PLIC、CSR、trap handler 都稳定后，再逐步引入真正的中断处理路径。

### 6. 每一步都要可运行、可测试、可回退

MiniOS 不是一次性写完的。每一阶段都必须有明确验收标准：

- 能否构建？
- 能否加载？
- 能否运行？
- 能否输出？
- 能否触发异常？
- 能否进入 trap handler？
- 能否通过测试？

## 四、当前基础状态

根据当前 myCPU 项目状态，MiniOS 可以依赖以下已有基础：

### 1. CPU 核心

已有能力：

- PC 寄存器
- 32 个通用寄存器
- fetch / execute 流程
- load / store 封装
- 异常处理入口
- 当前特权级记录

后续 MiniOS 需要 CPU 继续支持：

- 正确跳转到内核入口地址
- 正确处理 trap 入口
- 正确执行 mret / sret
- 正确维护 mepc / sepc、mcause / scause、mtval / stval

---

### 2. DRAM 模型

已有能力：

- 128MB 物理内存
- 小端序读写
- 8/16/32/64 位访问
- 非法地址检测

后续 MiniOS 需要 DRAM 支持：

- 内核镜像加载
- 栈空间分配
- 简单堆区规划
- 后期页分配器测试

---

### 3. Bus 总线

当前 Bus 主要负责 DRAM 路由。

后续 MiniOS 的第一个关键前置任务是：

> 将 UART、CLINT、PLIC 接入 Bus 的 MMIO 地址路由。

没有 MMIO 路由，OS 无法通过统一地址空间访问外设，也就无法真正完成串口输出、定时器中断和外部中断处理。

---

### 4. CSR 与异常模块

已有能力：

- CSR 地址槽
- M/S 模式常用寄存器
- 部分 M/S 模式寄存器映射
- 异常类型建模
- 异常上下文保存

后续 MiniOS 需要重点验证：

- mtvec / stvec 是否正确决定 trap 入口
- mepc / sepc 是否正确保存异常前 PC
- mcause / scause 是否正确记录异常或中断原因
- mtval / stval 是否正确记录异常附加信息
- mstatus / sstatus 的中断使能位是否能正确流转
- mret / sret 是否能正确恢复执行流

---

### 5. UART / CLINT / PLIC

当前 UART、CLINT、PLIC 已有独立模型和测试，但尚未完全进入系统运行路径。

MiniOS 后续需要让这些外设从“独立可测”变成“系统可用”：

- UART：用于内核日志输出
- CLINT：用于机器定时器中断
- PLIC：用于外部中断管理

## 阶段 2：UART 输出与内核日志

### 目标

让 MiniOS 能够通过 UART 输出字符，实现最小 `printk`。

### 核心意义

### 前置条件

- Bus 已经支持 UART MMIO 路由

- 内核可以访问 UART 地址

### 任务

- 将 UART 接入 Bus
- 在内核中实现 `uart_putc`
- 在内核中实现 `uart_puts`
- 实现最小 `printk`
- 让 `kernel_main` 输出启动信息

### 示例目标输出

```text
MiniOS booting...
Hello from kernel!
````

### 验收标准

* 裸机内核可以通过 UART 输出字符串
* 输出路径经过 CPU → Bus → UART
* UART 不只是单元测试可用，而是在系统运行路径中可用

### 推荐产出

* `os/kernel/uart.c`
* `os/kernel/uart.h`
* `os/kernel/printk.c`
* `docs/UART输出路径说明.md`

## 阶段 4：机器定时器与 CLINT

### 目标

让 MiniOS 支持机器定时器中断，为后续调度器做准备。

### 核心问题

* CLINT 的 mtime / mtimecmp 如何工作？

### 前置条件

### 验收标准

* ra / sp / callee-saved registers 如何保存？
* yield 如何切换任务？

### 任务

* 定义 task_struct
* 为每个任务分配内核栈
* 实现 context_switch 汇编
* 实现 yield
* 创建两个简单任务交替输出

### 验收标准

* 两个任务可以交替运行
* 每个任务拥有独立栈
* 上下文切换后寄存器状态不混乱
* UART 输出顺序可观察

### 推荐产出

* `os/kernel/task.c`

### 验收标准

* 开启分页前后内核仍可运行
* 虚拟地址可以正确映射到物理地址
* 非法权限访问能触发页异常
* satp 切换后地址翻译符合预期

## 六、OS 部分的最小可行目标

为了避免目标过大，MiniOS 的第一阶段不要定义成“做一个操作系统”。

更合理的第一版目标是：

> 在 myCPU 模拟器上运行一个裸机 MiniOS 内核，能够初始化栈，通过 UART 输出启动信息，并能处理一次 ecall 或非法指令异常。

第一版 MiniOS 验收标准：

* 模拟器能加载 MiniOS 二进制
* PC 从内核入口开始执行
* 栈指针正确设置
* UART 能输出 `MiniOS booting...`
* 能主动触发一次异常
* trap handler 能打印异常原因
* 系统不会无意义崩溃

这个目标完成后，项目就从“CPU 模拟器”正式进入“系统软件平台”。

## 八、不要现在做的事情

当前阶段暂时不要做：

* 完整进程模型
* 文件系统
* 用户态程序加载
* ELF loader
* 虚拟内存
* TLB
* Cache
* 多核
* 复杂设备驱动
* 完整 PLIC 优先级仲裁
* 完整类 Linux 系统调用表

这些内容可以作为长期方向，但不是当前 P0。

现在的重点是：

> 让一个最小内核真正跑在你的模拟器上。

### 每次开发只推进一个闭环

一个合格的小闭环包括：

1. 明确目标
2. 修改少量文件
3. 能构建
4. 能运行
5. 有测试或可观察输出
6. 写开发日志

# MiniOS 总规划补充（v1.1）：阶段 0.5 与关键工程约束

## 一、为什么需要补充阶段 0.5

原规划将 MiniOS 路线拆分为：

- 裸机程序加载
- UART 输出
- Trap 处理
- 定时器中断
- 内存管理
- 调度
- 系统调用
- S 模式
- 虚拟内存

这个方向是正确的，但在真正开始阶段 1 之前，还需要补充一个"**阶段 0.5**"。

阶段 0.5 的目标不是写 OS 功能，而是补齐 MiniOS 能够启动和调试的几个**关键前置条件**：

1. Bus MMIO 路由
2. 内存布局与链接脚本规划
3. CSR 访问抽象层
4. 指令集覆盖检查
5. 测试与回归策略

如果没有这些前置工作，后续即使写出了 `start.S`、`kernel_main` 和 `uart_puts`，也可能因为总线不通、链接地址错误、CSR 硬编码、指令未实现等问题陷入难以定位的调试泥潭。

因此，**阶段 0.5 是 CPU 模拟器项目向 MiniOS 项目过渡的工程桥梁**。

---

## 二、阶段 0.5：MiniOS 前置工程化改造

### 0.5.1 Bus MMIO 路由

#### 目标

让 Bus 不再只负责 DRAM 访问，而是成为统一地址空间的访问入口。

后续 MiniOS 访问 UART、CLINT、PLIC 时，不应该直接调用外设对象，而应该通过正常的 load/store 指令访问对应 MMIO 地址，再由 Bus 路由到具体设备。

#### 地址路由原则

Bus 应当根据访问地址判断目标设备：

```text
地址落在 DRAM 范围  → 路由到 DRAM
地址落在 UART 范围  → 路由到 UART
地址落在 CLINT 范围 → 路由到 CLINT
地址落在 PLIC 范围  → 路由到 PLIC
其他地址           → 产生访存异常
```

#### 为什么这是第一瓶颈

MiniOS 早期最重要的调试能力是 UART 输出。

如果 UART 没有接入 Bus，那么即使内核代码成功加载，内核也无法通过统一内存地址访问串口寄存器，最终只能靠单步调试或日志猜测运行状态。

因此，**Bus MMIO 路由不应该等到 UART 阶段才做，而应该和裸机加载同步推进**。

#### 最小任务

- 在 Bus 中加入 UART 地址范围判断
- 保持 DRAM 路由不变
- 非法地址继续抛出原有访存异常
- 为 UART MMIO 路由补充 BusTest
- 暂时不实现完整中断行为

#### 验收标准

- CPU 通过 `store` 指令写 UART MMIO 地址时，字符能够输出
- BusTest 覆盖 DRAM、UART、非法地址三类路径
- 原有 DRAM/CPU/Instruction 测试不退化

---

### 0.5.2 内存布局与链接脚本提前规划

#### 目标

在 MiniOS 启动前，先明确内核在 DRAM 中的布局，避免后续内存管理阶段大范围返工。

#### 推荐内存布局

```
DRAM_BASE = 0x80000000

0x80000000
├── .text       内核代码段
├── .rodata     只读数据段
├── .data       已初始化全局数据
├── .bss        未初始化全局数据
├── kernel_end  内核镜像结束位置
├── stack       启动栈区域
└── heap/page   后续物理页分配器管理区域
```

#### linker.ld 需要提前暴露的符号

建议在链接脚本中提前定义：

| 符号 | 含义 |
|------|------|
| `_kernel_start` | 内核起始地址 |
| `_text_start` / `_text_end` | 代码段边界 |
| `_rodata_start` / `_rodata_end` | 只读数据段边界 |
| `_data_start` / `_data_end` | 已初始化数据段边界 |
| `_bss_start` / `_bss_end` | BSS 段边界（启动阶段清零用） |
| `_kernel_end` | 内核镜像结束地址（物理页分配器起点） |
| `_stack_bottom` / `_stack_top` | 启动栈边界 |

#### 为什么要提前做

阶段 5 的物理内存管理需要知道：

- 哪些内存属于内核代码
- 哪些内存属于内核数据
- 哪些内存属于栈
- 哪些内存可以被分配

如果前期链接脚本不规范，后期实现 `kalloc` / `kfree` 时就会面临地址边界不清的问题。

#### 验收标准

- `kernel.elf` 的各段地址符合 DRAM_BASE 规划
- `objdump -h kernel.elf` 能看到清晰段布局
- `_stack_top` 能被 `start.S` 引用
- `_kernel_end` 能被后续内存管理模块引用

---

### 0.5.3 CSR 访问抽象层

#### 目标

避免 MiniOS 早期代码大量硬编码 M 模式 CSR，降低后续从 M 模式切换到 S 模式的重写成本。

#### 背景

早期 MiniOS 可以运行在 M 模式下，因此会使用：

- `mtvec` / `mepc` / `mcause` / `mtval`
- `mstatus` / `mie` / `mip`

但后续如果切换到 S 模式，就会使用：

- `stvec` / `sepc` / `scause` / `stval`
- `sstatus` / `sie` / `sip`

如果所有代码都直接写死 M 模式 CSR，那么阶段 9 切换到 S 模式时会产生大量重复修改。

#### 推荐做法

在 OS 代码中增加一个轻量 CSR 抽象层，例如：

```c
// os/include/csr.h

static inline uint64_t read_csr_mcause(void);
static inline void write_csr_mtvec(uint64_t value);
static inline uint64_t read_csr_mepc(void);
static inline void write_csr_mepc(uint64_t value);
```

或者使用更高层的宏命名：

```c
// 阶段 3-8（M 模式）
#define TRAP_VEC_REG     mtvec
#define TRAP_EPC_REG     mepc
#define TRAP_CAUSE_REG   mcause
#define TRAP_TVAL_REG    mtval
#define TRAP_STATUS_REG  mstatus

// 阶段 9 切换到 S 模式时只需改为：
#define TRAP_VEC_REG     stvec
#define TRAP_EPC_REG     sepc
#define TRAP_CAUSE_REG   scause
#define TRAP_TVAL_REG    stval
#define TRAP_STATUS_REG  sstatus
```

#### 原则

- 抽象层只做轻量封装，不要一开始设计复杂的权限模式框架
- 当前目标是：减少硬编码、统一 CSR 读写入口、为 M/S 模式切换预留空间

#### 验收标准

- trap 相关代码不直接散落大量裸 CSR 名称
- M 模式 trap handler 可以正常工作
- 后续切换 S 模式时不需要重写全部 trap 代码

### 0.5.4 指令集覆盖检查

#### 目标

确保 MiniOS 编译出来的内核二进制只包含当前模拟器能够执行的指令。

#### 背景

即使内核 C 代码非常简单，编译器也可能生成模拟器尚未实现的指令。

可能出现的问题链：

1. 内核能编译 ✓
2. 内核能加载 ✓
3. 但运行到某条未实现指令时触发 `IllegalInstruction` ✗
4. 调试时误以为是启动逻辑、链接脚本或内存错误

因此，**在运行 MiniOS 前，必须检查内核实际用到的指令**。

#### 推荐命令

```bash
riscv64-unknown-elf-objdump -d kernel.elf > kernel.asm
```

然后检查指令助记符，与模拟器 `InstructionExecutor` 中已实现指令列表对比。

#### 检查内容

- 内核用了哪些指令
- 这些指令是否已经在 InstructionExecutor 中实现
- 这些指令是否已经进入分派表
- 这些指令是否已有测试
- 是否出现压缩指令（C 扩展）
- 是否出现乘除法（M 扩展）
- 是否出现未支持的伪指令展开结果

#### 编译建议

早期尽量关闭复杂扩展：

```makefile
CFLAGS = -march=rv64i -mabi=lp64 -ffreestanding -nostdlib -nostartfiles
```

如果当前模拟器尚未支持压缩指令，不要使用 `rv64gc`。

#### 验收标准

- `kernel.asm` 中出现的真实指令都被模拟器支持
- 未支持指令有明确清单
- 运行失败时能区分是"指令未实现"还是"OS 逻辑错误"

### 0.5.5 测试与回归策略

#### 目标

让 MiniOS 开发不是靠感觉推进，而是每个阶段都有验证方式。

#### 测试分层

MiniOS 后续建议保留三类测试：

**1. 单元测试**（Google Test）

用于测试模拟器内部模块：

- DRAM / Bus / CSR / Exception
- UART / CLINT / PLIC
- InstructionExecutor

**2. 裸机集成测试**

用于测试一段裸机程序在模拟器中是否按预期运行：

- 执行简单算术
- 写 UART
- 触发 `ecall`
- 触发非法指令
- 触发 timer interrupt

**3. OS 行为测试**

用于测试 MiniOS 的可观察行为：

- 启动后输出 `MiniOS booting...`
- 触发异常后输出 `trap: mcause = ...`
- 定时器中断输出 tick
- 两个任务交替输出

#### 对比测试

后续可以考虑使用 Spike 或 QEMU 作为参考实现，对比同一个裸机程序的行为。

对比内容可以包括：

- PC 变化
- 寄存器结果
- CSR 值
- 异常原因
- 内存结果

早期不强制引入 Spike，但应当预留这个方向。

#### 回归要求

每完成一个新阶段，都应当运行：

```bash
cmake --build build_wsl
ctest --test-dir build_wsl --output-on-failure
```

如果引入 OS 构建系统，还应运行：

```bash
cd os
make clean
make
```

以及对应模拟器运行命令。

#### 验收标准

- 新功能有新测试或可观察输出
- 老测试不退化
- 每个阶段有对应日志
- 每次 commit 前都能复现构建和运行结果

## 四、当前最小闭环目标修订

原先的一句话目标是：

> 在已有 RISC-V 模拟器基础上，打通"裸机内核加载 → UART 输出 → Trap 处理"这条最小 OS 运行链路。

现在修订为：

> 在已有 RISC-V 模拟器基础上，**先补齐 Bus MMIO、内存布局、CSR 抽象和指令覆盖检查**，再打通"裸机内核加载 → UART 输出 → Trap 处理"这条最小 OS 运行链路。

这个修订后的目标更加稳妥。

因为 MiniOS 的第一条生命线不是调度器，也不是虚拟内存，而是：

1. `kernel.bin` 能被加载
2. PC 能跳到入口
3. 栈能正常设置
4. UART 能说话
5. 异常能进入 trap
6. 测试能回归

只要这条链路打通，MiniOS 才算真正开始。
