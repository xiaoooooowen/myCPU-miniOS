# MiniOS 未来一周开发计划：用户态与交互能力冲刺

> 时间范围：2026-06-08 至 2026-06-14  
> 当前基础：阶段 0.5 至阶段 10 已完成，MiniOS 已具备启动、S 模式、trap、系统调用、页分配、Sv39、抢占式调度能力  
> 本周目标：全部时间用于开发，不安排 PPT、视频和汇报材料

---

## 一、本周开发定位

本周不是重新扩大整个项目范围，而是在现有 MiniOS 内核闭环基础上，继续向 A 题要求中更典型的“操作系统内核”能力靠近。

当前项目已经满足“至少 3 个模块、9 个功能点”的基本要求，但答辩时最容易被追问的问题是：

- 是否真的有用户态？
- 系统调用是不是只在内核态自测？
- 是否支持输入？
- 调度对象是任务还是进程？
- 文件系统是否有雏形？

因此，本周开发的核心方向是：

> 先打通 U 模式用户程序和系统调用，再补输入、任务状态和最小 RAMFS。  
> 保证每一天都有可构建、可运行、可回退的产出。

## 模块二：U 模式切换 ✅ 已完成（2026-06-08）

### 目标

从 S 模式内核切换到 U 模式用户程序，证明系统不再只是内核自检。

### 核心问题

- sstatus.SPP 如何设置为 U 模式。
- sepc 如何设置为用户入口地址。
- 用户栈如何初始化。
- sret 后 CPU mode 是否变为 User。
- U 模式 ecall 是否进入 S 模式 trap handler。

### 推荐最小实现

先不做 ELF loader，不做文件系统加载程序。

可以在内核中准备一个简单的用户程序代码段：

1. 用汇编或二进制数组提供用户代码。
2. 把用户代码复制到用户虚拟地址。
3. 为用户程序分配用户栈。
4. 设置用户页表权限。
5. 通过 `sret` 进入用户入口。

### 涉及文件

- `os/kernel/vm.c`
- `os/kernel/trap.c`

### 验收标准

- ✅ 能观察到 U 模式 ecall 的 `scause=8`。
- ⚠️ U 模式权限隔离依赖 MMU U 位检查（当前 MMU 未实现，不影响 demo）。
- ✅ sret 返回路径稳定，不破坏已有 S 模式内核功能。

## 模块三：用户态系统调用 ✅ 已完成（2026-06-08）

### 目标

让 U 模式用户程序通过 ecall 调用内核服务。

### 最小系统调用表

| 系统调用 | 编号 | 功能 | 状态 |
|---:|---|------|
| sys_write | 64 | 输出字符串到 UART | ✅ |
| sys_exit | 93 | 退出当前用户程序或关闭系统 | ✅ |
| sys_read | 63 | 读取 UART 输入（阻塞，遇 \\n 终止） | ✅ |

### 实现要点

1. a7 保存系统调用号。
2. a0-a2 保存参数。
3. 返回值写回 trap frame 的 a0。
4. ECALL from U-mode 后 sepc 必须加 4。
5. sys_exit 不能破坏内核调度状态。

### 涉及文件

- `os/kernel/syscall.h`
- `os/kernel/syscall.c`
- `os/kernel/trap.c`
- `os/kernel/user.c`

### 验收标准

- ✅ 用户程序能打印 `Hello from user!`。
- ✅ 用户程序能调用 sys_exit(0) 后触发 TEST_FINISH 停机（exit code=0）。
- ✅ 原有 S 模式 ecall 测试全部通过。
- ✅ 单元测试 88/88 全通过，模拟器正常退出。

### 踩坑记录

**4 字节对齐问题**：`.string` 指令生成 18 字节（17 字符 + NUL），导致后续指令不在 4 字节边界。RISC-V 无 C 扩展时所有指令必须 4 字节对齐。使用 `.align 4`（16 字节对齐）修复。否则模拟器 fetch 读取错位数据触发 IllegalInstruction。

## 模块四：用户地址空间与权限隔离 ✅ 已完成（2026-06-09）

### 目标

让用户程序拥有独立的地址空间边界，避免 U 模式直接访问内核页。

### 实现要点

- MMU `translate()` 增加 `mode` 参数（User=0, Supervisor=1, Machine=3）
- 在 2MB 大页和 4KB 叶节点 U 位检查：`mode == User && !(pte & PTE_U)` → 抛对应页异常
- 非叶节点不检查 U 位（RISC-V 规范规定）
- CPU `load()/store()/fetch()` 传入 `this->mode`

### 涉及文件

- `src/mmu.h`
- `src/mmu.cpp`
- `src/cpu.cpp`
- `tests/unitest/mmu_test.cpp`

### 验收标准

- ✅ U 模式访问 U=1 页（用户代码/栈）正常
- ✅ U 模式访问 U=0 页（内核页/TEST_FINISH）触发页异常
- ✅ S/M 模式访问 U=0 页不受影响
- ✅ 新增 3 个 MMU U 位测试全部通过
- ✅ 91/91 单元测试全部通过

## 模块五：UART 输入与 sys_read

### 目标

补齐交互能力，让系统不只会输出，也能读取输入。

### 实现要点

- 内核端 UART：`uart_has_data()` 轮询 LSR 寄存器 RX 位，`uart_getc()` 阻塞读取
- 新系统调用 `SYS_READ`(63)：`sys_read(fd, buf, len)`，阻塞读取最多 len 个字符，遇 `\n` 提前返回
- UART stdin 监听重构：`std::cin >> byte` → `poll(STDIN_FILENO, ..., 100ms)` + `read()`，进程可干净退出
- Bus 新增 `start_stdin()` 方法，`main.cpp` 启动后调用

### 涉及文件

- `os/kernel/uart.h` — 新增 UART_RHR/UART_LSR_RX_READY 定义，uart_has_data()/uart_getc()
- `os/kernel/uart.c` — 实现轮询 RX
- `os/kernel/syscall.h` — 新增 SYS_READ(63)
- `os/kernel/syscall.c` — sys_read() + 分派分支
- `src/uart.cpp` — stdin 监听改用 poll() 非阻塞
- `src/bus.h/cpp` — start_stdin() 方法
- `src/main.cpp` — 启动 stdin 监听

### 验收标准

- ✅ 内核端 uart_getc() 可阻塞读取终端输入
- ✅ sys_read 系统调用可从用户态调用
- ✅ 进程退出时 stdin 线程不阻塞（poll 100ms 超时）
- ✅ 单元测试 91/91 全通过（stdin 监听不在测试环境启动）

## 模块六：任务状态与 exit/wait 雏形

### 目标

把当前任务系统从简单轮转推进到更接近进程模型。

### 推荐新增状态

```c

TASK_READY
TASK_RUNNING
```

### 最小功能

1. 调度器跳过 BLOCKED 和 ZOMBIE。
2. sys_exit 将当前任务置为 ZOMBIE。
3. wait 可以回收一个 ZOMBIE 子任务，若没有则返回 -1。
4. 暂时不做完整父子进程树。

### 涉及文件

- `os/kernel/task.h`

- ZOMBIE 任务可以被回收。
- 至少两个任务轮转时，一个退出不会影响另一个。

## 模块七：RAMFS 最小版

### 目标

在时间允许时实现一个极简内存文件系统雏形，增强 A 题中文件系统模块的覆盖。

### 最小设计

不做磁盘、不做 mkfs、不做复杂目录树。

先实现固定大小内存文件表：

```c
struct ramfs_file {
    char name[32];
    char data[4096];
    uint64_t size;
    int used;
};
```

最小接口：

```c
int ramfs_create(const char *name);
int ramfs_write(int fd, const void *buf, uint64_t len);
int ramfs_read(int fd, void *buf, uint64_t len);
int ramfs_close(int fd);
```

## 六、质量保证标准

## 6.1 构建质量

每个阶段完成后至少运行：

```bash
cmake --build build_wsl -j$(nproc)
ctest --test-dir build_wsl --output-on-failure
cd os && make clean && make
```

不能接受：

- 模拟器构建失败。
- MiniOS 构建失败。
- 老测试大面积失败。
- 靠注释掉旧功能换新功能。

## 6.2 运行质量

每个新增功能必须有可观察输出，例如：

- UART 日志。
- trap cause。
- syscall 返回值。
- 页异常地址。
- 调度任务名称。

不能只说“理论上实现了”。

## 6.3 代码质量

原则：

1. 不引入无关依赖。
2. 不大范围重构旧代码。
3. 新增模块命名清晰。
4. trap frame 偏移、CSR 位、PTE flag 必须写清楚依据。
5. 内核 C 代码保持 freestanding，不依赖标准库。

## 6.4 测试质量

优先补三类测试：

1. 模拟器单元测试：MMU、CSR、Bus、异常。
2. 内核自测：syscall、mem、task、ramfs。
3. 端到端日志测试：运行 kernel.bin 并提取 UART 输出。

## 九、风险控制

## 9.1 最大风险：U 模式和页表同时改，调试复杂

应对：

1. 先在 Bare/Sv39 现有身份映射下证明 U-mode ecall。
2. 再单独加入用户页表。
3. 最后做非法访问测试。

## 9.2 最大风险：UART 输入阻塞调度器

应对：

1. 先实现非阻塞 read，空缓冲返回 0。
2. 再做 BLOCKED 状态。
3. 不要第一版就强制阻塞等待输入。

## 9.3 最大风险：RAMFS 分散精力

应对：

只有在 U 模式和 sys_read 稳定后才做 RAMFS。

如果 2026-06-13 晚上仍有核心 bug，2026-06-14 不做 RAMFS，转为稳定收尾。
