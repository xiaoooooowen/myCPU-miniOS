### 任务一：0.5.1 Bus MMIO 路由

**这是什么问题：**

当前 [bus.cpp](file:///home/xiaowen/projects/mycpu/src/bus.cpp#L13-L27) 的 `load/store` 逻辑很简单：

```cpp
if (addr >= DRAM_BASE && addr <= DRAM_END) {
    return dram.load(addr, size);
}
throw Exception(ExceptionType::LoadAccessFault, addr);
```

也就是说，CPU 想访问 UART 地址（`0x10000000`）写一个字符时，Bus 直接抛异常。MiniOS 的内核代码即使写对了 `*(volatile uint8_t*)0x10000000 = 'A'`，也会因为 Bus 不认这个地址而崩溃。

**你需要做的事情：**

在 Bus 类中加入 UART、CLINT、PLIC 的地址路由判断：

```
DRAM 范围  0x80000000 - 0x87FFFFFF  → dram.load/store
UART 范围  0x10000000 - 0x1000000F  → uart.load/store
CLINT 范围 0x02000000 - 0x0200BFFF  → clint.load/store
PLIC 范围  0x0C000000 - 0x0FFFFFFF  → plic.load/store
其他                               → 访存异常
```

**理论连接：**

这是典型的 **Memory-Mapped I/O（MMIO）**。课本讲 I/O 时有三种方式：Programmed I/O、Interrupt-driven I/O、DMA。MMIO 就是 Programmed I/O 的具体实现——CPU 用 `load/store` 指令访问外设寄存器，就像访问内存一样。RISC-V 没有独立的 `in/out` 指令（x86 有），完全依赖 MMIO。

**你能独立完成吗：** ✅ **可以，这是最适合你动手的任务。**

- 三个外设类（[Uart](file:///home/xiaowen/projects/mycpu/src/uart.h)、[Clint](file:///home/xiaowen/projects/mycpu/src/clint.h)、[Plic](file:///home/xiaowen/projects/mycpu/src/plic.h)）已经写好了 `load(addr, size)` 和 `store(addr, size, value)` 接口
- 地址常量大部分已在 [param.h](file:///home/xiaowen/projects/mycpu/src/param.h) 中定义
- 已有 [bus\_test.cpp](file:///home/xiaowen/projects/mycpu/tests/unitest/bus_test.cpp) 供你模仿扩展
- 这是一个改动范围明确、验收标准清晰的任务

**需要处理的细节（工程挑战）：**

1. Bus 当前只构造 `dram`，需要加入 `uart/clint/plic` 成员
2. UART/CLINT/PLIC 的地址范围常量需要补充或复用 param.h 中已有的（UART 的 BASE 地址需要确认）
3. 测试要覆盖"通过 Bus 写 UART 地址，字符输出到 stdout"

### 任务二：0.5.2 内存布局与链接脚本

**这是什么问题：**

当你用 GCC 编译一个 C 程序时，链接器（ld）需要知道把代码（`.text`）、只读数据（`.rodata`）、全局变量（`.data`）、未初始化数据（`.bss`）放到内存的什么位置。对于裸机程序（没有 OS），你需要自己写 **链接脚本（linker script）** 告诉链接器怎么做。

文档规划的内存布局：

```
0x80000000  ← DRAM_BASE，也是内核的加载地址
├── .text       内核代码
├── .rodata     只读数据（如字符串常量）
├── .data       已初始化的全局变量
├── .bss        未初始化的全局变量（启动时清零）
├── kernel_end  这些段结束的位置
├── stack       栈区域
└── heap/page   未来物理页分配器用的区域
```

**理论连接：**

- **ELF 格式**：`.text/.data/.bss` 这些段名是 ELF（Executable and Linkable Format）标准的一部分，不是 GCC 随便起的
- **BSS 段**为什么不在可执行文件中占空间？因为它全是 0，ELF 文件只记录 BSS 段的大小而不存储实际数据，加载时才分配并清零——这是一个经典面试题
- **链接地址 vs 加载地址**：裸机程序的链接地址必须等于物理内存的起始地址（对 MiniOS 就是 `0x80000000`），否则绝对跳转会飞到错误位置

**你能独立完成吗：** ⚠️ **可以写初稿，但需要 AI 辅助审查。**

- 链接脚本的语法（GNU ld script）比较晦涩，第一次写容易出错
- 但是**理解它的思想**比你手写完美脚本更重要
- 用 `objdump -h kernel.elf` 验证输出，这是你需要掌握的调试技能

**你能承担的具体工作：**

- 理解 `.text/.data/.bss` 各自的含义和加载行为
- 手写第一版 `os/linker.ld`
- 用 `riscv64-unknown-elf-objdump -h kernel.elf` 检查输出的段布局
- 验证 `_kernel_end` 等符号是否正确暴露

### 任务三：0.5.3 CSR 访问抽象层

**这是什么问题：**

RISC-V 中有两种相似的异常处理寄存器组：

- M 模式用：`mtvec / mepc / mcause / mtval / mstatus / mie / mip`
- S 模式用：`stvec / sepc / scause / stval / sstatus / sie / sip`

当前模拟器的 CSR 操作通过直接读写 `csr.csrs[MTVEC]` 这样的数组。如果我们的 MiniOS 在阶段 1-8 写了很多 `csr.csrs[MTVEC]` 这样的代码，阶段 9 切换到 S 模式时要全部改一遍。用一个宏/内联函数做一层薄封装就能避免这个问题。

**理论连接：**

- **RISC-V 特权级**（Privilege Levels）：M > S > U。允许硬件在同一个 trap handler 入口对不同模式使用不同的 CSR
- **软件工程原则**：这是"封装变化"的经典案例。不预先做过度抽象，但也不把"未来一定会变的东西"硬编码到所有调用点

**你能独立完成吗：** ✅ **完全可以，这是纯 C 宏/内联函数的工作。**

这里的难度不在编程，而在于理解"为什么要这么做"。理解了"封装变化点"这个思想，这关就过了。

**你能承担的具体工作：**

- 设计 `os/include/csr.h` 的宏定义方案
- 决定是用 `#define` 宏还是 `static inline` 函数
- 为每个 trap 相关的 CSR 定义统一的访问入口

### 任务四：0.5.4 指令集覆盖检查

**这是什么问题：**

你的 RISC-V 模拟器实现了约 40 条指令，但 GCC 编译器可能生成你还没实现的指令（比如 `SUB`、`BNE`、`BLT`）。如果编译器生成的指令在你的分派表中找不到，模拟器就抛 `IllegalInstruction` 异常。

已有的 [CURRENT\_STATUS.md](file:///home/xiaowen/projects/mycpu/docs/CURRENT_STATUS.md#L101-L138) 已经做了非常详尽的差距分析。

**理论连接：**

- **RISC-V 指令编码**：每条指令由 opcode（低 7 位）+ funct3 + funct7 唯一标识
- **编译器与 ISA 的关系**：`-march=rv64i` 告诉编译器"只能生成 RV64I 这个子集的指令"

**你能独立完成吗：** ⚠️ **主要是"审核已有分析"而非"从零开始"。**

CURRENT\_STATUS.md 已经把缺失的分派表条目列得很清楚了（[第 116-138 行](file:///home/xiaowen/projects/mycpu/docs/CURRENT_STATUS.md#L116-L138)），你的工作是：

- **审核**：验证这些分析是否准确
- **优先级排序**：哪些指令 MiniOS 一定会用到，必须先补？哪些可以延后？
- **补充分派表条目**：这虽然看起来是机械工作，但需要你理解 instruction decoding 的 `(opcode, funct3, funct7)` 三元组映射机制

**你能承担的具体工作：**

- 对照指令表，逐个确认分派表缺失项
- 排序：`SUB`/`BNE`/`BLT`/`ECALL` 是 P0，其他可以排后
- 补全分派表条目并编写对应测试
